# v2.7 — Pro Operator

Three Pro features that together make ZPan safe to open up to the public: audit log (what happened), webhooks (tell my other systems), and anti-abuse tooling for the `open` registration mode (which was Pro-gated in v2.6).

All three are local to ZPan — no cloud service dependency.

## Major Features

### Audit Log

- **User actions logged**: upload, download, delete, move, rename, share created / revoked, login, password change
- **Admin actions logged**: user create / disable / delete, role change, storage config change, branding change, license activation
- **Log entry fields**: timestamp, actor (user / admin / system), action, subject (file / user / config), IP, user-agent, result (success / failure)
- **Storage** — dedicated `audit_log` table, indexed on (actor, timestamp) and (action, timestamp)
- **Retention** — configurable: 30 / 90 / 365 days / forever. Default 90.
- **Admin UI** — filterable list, date range, actor search, action filter
- **Export** — CSV and JSONL download for compliance / ingestion into external SIEM
- **No public API for audit log in v2.7** — read access via admin UI only. API access is a Team-tier addition later.

### Webhook Notifications

- **Events**: `file.uploaded`, `file.deleted`, `share.created`, `share.accessed`, `share.revoked`, `user.created`, `quota.warning`
- **Endpoint configuration** — admin adds webhook URL, picks subscribed events, gets a signing secret
- **Payload signing** — HMAC-SHA256, header `X-ZPan-Signature: sha256=...`
- **Delivery**:
  - Async queue, fire-and-forget from main request path
  - Retry on 5xx / network error: 1m, 5m, 30m, 2h, 12h (5 attempts)
  - 4xx → no retry, marked failed
- **Delivery log** — per-event record: payload, response status, latency, retry count
- **Admin UI** — list endpoints, test-fire button (sends a sample payload), inspect delivery history

### Anti-Abuse Stack for Open Registration

The `open` registration mode itself became Pro-gated in v2.6. This version ships the anti-abuse tooling that makes `open` mode safe to turn on — without it, Pro users who enable `open` must roll their own protection at the CDN layer.

Active only when registration mode is `open`:

- **Email verification** — double opt-in, account inactive until verified
- **Captcha** — Turnstile (CF) / hCaptcha on signup form, configurable provider
- **Rate limiting** — IP-based signup throttle (5/hour/IP default), admin-configurable
- **Disposable email block** — bundled list of known disposable domains, admin can add to allowlist
- **Username uniqueness + reserved list** — reject admin / root / zpan / system / ...
- **Admin moderation** — new signups visible in user list, admin can bulk-disable / delete suspicious accounts

Gated under `requireFeature('open-registration')` — the same feature flag as the mode toggle itself, so the whole operator-registration toolset is one on-or-off thing from a licensing perspective.

## Non-goals for v2.7

- **Advanced RBAC / custom roles** — v2.8
- **Quota per user** — v2.8 (part of multi-tenant)
- **Public API access to audit log** — Team tier later
- **Intrusion detection / anomaly detection on audit stream** — way later

## User Scenarios

**SaaS operator running ZPan for customer file transfer:**
> I opened public registration so my customers can sign up themselves. Turnstile and email verification keep bots out. Audit log tells me who downloaded what, and webhooks push file events into our existing Slack channel.

**Compliance-sensitive internal deployment:**
> HR stores confidential docs in ZPan. I don't need open registration, but I do need audit logs for quarterly compliance review. I export last quarter as CSV, done.

**Indie developer with a Discord community:**
> Webhooks fire on every upload to a Discord channel so my community sees what's new without me announcing it.
